# Skill: 阻塞判定与上报

> 专项流程:同一路径失败 ≥2 次时的分类、上报、升级。
> 上位规则:[AGENTS.md](../../AGENTS.md) §3(上下文管理)、§5(硬规则)。

## 触发条件

- 同一执行路径失败 ≥2 次
- 环境自检或自动拉起失败(从 [skill-task-intake.md](skill-task-intake.md) 转入)
- 浏览器与命令行验证结果矛盾
- 执行超 2 轮仍未闭环

## 输入前提

- 已有至少两轮执行记录可复盘
- 已知失败现象的命令 / 日志 / 错误信息
- 当前环境口径明确(dev / dev:ai / dev:cc)

## 执行步骤

### A. 阻塞分类

先选定阻塞类别,不同类别的根因和排查方向不同:

| 类别 | 典型表现 |
|------|---------|
| **H5 前端** | 页面报错、状态未更新、组件渲染异常 |
| **API 后端** | 接口返回错、知识匹配失败、AI 调用异常 |
| **Admin 前端** | 管理页报错、表单提交失败、数据列表空 |
| **环境变量** | env 未读到、默认值缺失、端口读错 |
| **CORS** | 预检失败、`Origin` 不在白名单、浏览器阻断 |
| **工具链** | npm install 失败、TypeScript 编译错、Vite 构建错 |
| **任务边界** | 任务说明不清、跨阶段需求、受保护文件需修改 |
| **Git 规范** | commit 格式、scope 选择、hook 失败 |

### B. 阻塞上报格式

必须一次性输出全部字段,不能省略:

```text
阻塞分类:[H5前端 / API后端 / Admin前端 / 环境变量 / CORS / 工具链 / 任务边界 / Git规范]
当前现象:[具体报错 / 不符合预期的行为]
已确认事实:[已经验证过的事实,避免重复排查]
最可能根因:[基于已确认事实的推断]
下一步最小验证动作:[单步可执行的最小验证,不要批量猜测]
```

### C. 症状速查表

| 现象 | 优先排查 |
|------|---------|
| **401** | `x-api-key` 未带 或 `VITE_API_SECRET ≠ API_SECRET` |
| **CORS 报错** | H5 端口不在 `CORS_ORIGINS` |
| **`matched` 永远 null** | 知识条目关键词未覆盖,先查 knowledge 数据与检索逻辑 |
| **`import.meta.env.VITE_*` 是 undefined** | 对应环境文件不存在 或 dev server 未重启 |
| **新路由 401 但 key 正确** | 忘记挂 `authMiddleware` |
| **明明已启动但健康检查失败** | 先核对当前环境口径(dev / dev:ai / dev:cc),再核对端口 |
| **浏览器失败但命令行成功** | 优先 CORS / Origin / 预检 / 前端鉴权 / 路由前缀,不是业务失败 |
| **命令行失败但服务曾启动成功** | 先复查端口监听、进程存活、环境口径、请求 Header |

### D. Windows 编码问题(独立类别)

命令行中文请求体异常时,**先排查终端编码,不要直接判断接口或 AI 返回异常**。处理方式:

- 优先使用 UTF-8 编码文件 + `curl --data-binary @file.json`
- 或使用 Unicode 转义 JSON(如 `\u6D4B\u8BD5`)
- PowerShell 和 bash 编码行为不同,注意区分
- **Windows 终端显示乱码 ≠ 接口一定有问题**,必须结合实际响应文件内容和浏览器链路判断

### E. 环境阻塞(从 task-intake 转入)

若触发原因是环境自检或自动拉起失败,按以下格式补充:

```text
环境阻塞详情:
- 本次环境口径:dev / dev:ai / dev:cc
- 失败步骤:端口监听 / 健康检查 / 自动拉起
- 尝试命令:[具体命令]
- 实际输出:[错误信息]
- 环境文件状态:[.env / .env.ai / .env.cc 是否存在]
```

### F. 升级路径

**同一路径最多尝试 2 次,失败后必须切换排查层级。** 升级方向:

1. **单文件层** → 多文件相互影响
2. **单端层** → 跨端联调
3. **代码层** → 环境 / 配置 / 工具链
4. **Codex 5.4 执行** → 升级至 Codex 5.5 接手
5. **执行层** → 升级至 ChatGPT 5.5 重新规划

切换层级前,在阻塞报告中明确标注:

```text
升级决策:
- 已尝试层级:[当前层]
- 失败次数:[N 次]
- 拟切换至:[目标层]
- 理由:[为什么当前层无法闭环]
```

**不升级的情况**:

- 阻塞分类为"任务边界"或"受保护文件" → 直接停止,等用户确认,不升级模型
- 阻塞涉及用户业务判断(如需求冲突) → 直接停止,等用户决策

升级的前提是"同类问题更强的模型能解决",不是所有阻塞都适合升级。

## 输出要求

必须输出:

- 阻塞分类(B 节格式)
- 症状速查参考(若表内命中对应行)
- 下一步最小验证动作(单步可执行)

若涉及环境阻塞 → 追加 E 节格式。若涉及升级 → 追加 F 节升级决策。

## 完成判定

满足任一:

- 最小验证动作执行后,根因已定位 → 回到对应 Skill 继续执行
- 同路径 2 次失败 → 升级决策已输出,切换至更高层级
- 阻塞涉及任务边界 / 受保护文件 / 跨阶段 → 停止并等待用户确认

**未完成的信号**:继续在同一层级无头绪尝试第 3 次 → 这是违反升级规则,必须立即停止并按 F 节升级。
