# Skill: Codex 执行任务

> 专项流程:Codex 开始执行任务时的代码修改、执行包推进、提交前自查。
> 上位规则:[AGENTS.md](../../AGENTS.md) §5(硬规则)、§6(阶段边界)。

## 触发条件

- Codex 开始执行代码修改任务
- 推进一个执行包(执行包 A / B / C)
- 做 bug 修复或联调补丁

## 输入前提

- [skill-task-intake.md](skill-task-intake.md) 已完成,环境自检通过
- 当前任务清单已读,任务范围已明确
- 已确认本次修改不触碰受保护文件(清单见 [AGENTS.md](../../AGENTS.md) §5.1)

## 执行步骤

### A. 任务颗粒度判定

默认目标:每一步尽量有效、低耦合、少错、便于复盘。

**可合并执行**:
- 同层、低耦合、无联调风险的纯新建 / 纯实现任务
- 同一文件内的连续小逻辑
- 同一闭环内的最小验证动作(编译检查 + 手动函数验证 + 回归检查)

**必须拆开执行**:
- 涉及 route / service / data / ai 分层边界的任务
- 涉及鉴权、参数校验、路由注册的任务
- 涉及环境切换、联调、CORS、回归验证的任务
- 涉及用户可见行为变化或接口契约风险的任务
- 涉及执行包 A / B / C 边界的任务

不确定时 → 优先拆开,不要为省步骤扩大风险。

### B. 代码修改执行

遵循 [AGENTS.md](../../AGENTS.md) §5.1 禁止事项 + §5.2 代码铁律。核心要点:

- 只改本次任务相关文件,不做"顺手优化"
- 非当前执行包范围的问题 → 先记录,不顺带修
- 新增依赖 / 环境变量 → 先告知用户,等待确认后再加
- 新增环境变量需同步 `.env.example` 与读取逻辑
- 文档任务不得顺手修改业务代码;业务任务不得顺手重写规范文档

### C. 执行包闭环推进

用户明确要求"执行包 / 功能包"时,默认按整包闭环推进:

1. 读文档与环境检查(skill-task-intake)
2. 连续修改相关文件
3. build / 联调 / 回归
4. 清理运行期副产物
5. 输出变更总结
6. 最后一次性提交(不单独为中间步骤提交)

**执行粒度 ≠ 提交粒度**:中间允许多次 build、联调、修正,但默认只在整包验收通过后 commit。临时兼容、调试探针、联调补丁不单独形成 commit,除非用户明确要求拆包。

### D. 临时兼容收口

- 新旧链路切换期间允许最小临时兼容,**必须在最近一个收口包中移除**,不能长期保留
- 前端 store / 页面不得长期绕过统一 API 封装直连后端;契约字段归一化应收敛在 API 层或服务层
- 若本轮引入桥接逻辑,输出中必须明确标注其是否已收口
- 未收口时不得直接表述为"完全闭环"

### E. 运行期文件清理

提交前默认执行 `git status --short`。核心规则:

- 运行期日志、`dist`、临时样本文件 **默认不进 commit**
- 提交前先恢复运行期文件到 `HEAD`
- 样本数据只能通过真实接口调用产生,**不能手工修改运行期 JSON**
- `questionLogs.json` / `missedQuestions.json` 在 P1 执行包 A 收口时移出 git tracking 并更新 `.gitignore`

### F. 提交前自查清单

提交前逐项确认:

- [ ] 引入新依赖? → 已告知用户并获授权
- [ ] 修改接口契约? → 已告知用户,破坏性变更已标注
- [ ] 新增环境变量? → 已同步 `.env.example`
- [ ] H5 `import.meta.env.VITE_*` 有默认值?
- [ ] API 新路由挂了 `authMiddleware`?
- [ ] 修改了受保护文件? → 授权已显式记录
- [ ] 本轮环境口径与验证命令一致?
- [ ] 已执行 `git status --short`?
- [ ] 运行时文件、`dist`、测试产物未进入 commit?
- [ ] 当前 commit 只做一类事?

## 输出要求

执行过程中输出:

- 任务颗粒度判定:可合并 / 必须拆开(标明依据)
- 修改文件清单(按执行顺序)
- 临时兼容是否已收口(若有)

收口前输出自查结果(以上清单全项勾选或说明原因)。

## 完成判定

同时满足:

- 所有自查项通过或有明确说明
- 临时兼容已收口 或 在执行摘要中显式标注为未收口
- 运行期文件已清理,`git status --short` 干净
- 可进入 [skill-acceptance-review.md](skill-acceptance-review.md) 做收口验收

任一项未通过 → 不得直接提交,需回到相应步骤或转 [skill-blocker-handling.md](skill-blocker-handling.md)。
