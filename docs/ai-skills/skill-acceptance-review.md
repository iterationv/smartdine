# Skill: 任务收口与验收

> 专项流程:任务执行完毕准备收口时的验收动作与摘要产出。
> 上位规则:[AGENTS.md](../../AGENTS.md) §3.3(执行摘要强制产出)、§8(输出格式要求)。

## 触发条件

- 代码改动完成,[skill-codex-execution.md](skill-codex-execution.md) 自查已通过
- 执行包所有条目已实现,准备阶段总验收
- 遗留问题收口包执行完毕

## 输入前提

- 本轮环境口径已明确(dev / dev:ai / dev:cc)
- 修改文件清单已整理
- 业务代码改动已自查通过

## 执行步骤

### A. 验收前置确认

验收前先核对,不默认假设:

- **真实页面路由**(不默认假设 `/admin/` 或其他前缀)
- **真实接口路由**(不默认假设 `/api/` 前缀)
- **端口**(与本轮环境口径一致)
- **鉴权方式**(`x-api-key` 是否需要携带)

错误态、空态、未命中态、无数据态这类"非自然发生"的验收项 → 如果任务说明未给出构造方式,应先补足最小构造方案再执行,不得跳过或脑补。

### B. 构造最小验证链路

按任务类型选择验证方式:

- **纯后端任务** → 命令行 curl 验证即可
- **前端联调任务** → 默认 **同时** 考虑浏览器验收和命令行验收,不能只用其中一种下结论
- **Admin 任务** → 默认同时验证命令行(后端数据正确性)和 Admin 页面(交互正确性),登录鉴权走 Admin 账号体系
- **浏览器失败 + 命令行成功** → 优先判定为 CORS / Origin / 预检 / 前端鉴权 / 路由前缀问题,不要直接认定业务失败(详见 [skill-blocker-handling.md](skill-blocker-handling.md))
- **命令行失败 + 服务曾启动成功** → 先复查端口监听、进程存活、环境口径、请求 Header,再判断代码问题

浏览器验收默认使用 Chrome;找不到时停止并告知用户。

### C. 跨任务回归判定

总体验收或跨任务回归时,明确区分:

1. **本轮直接验证**:本次任务实际跑过的验证项
2. **通过链路侧面确认**:未直接验证但通过上下游链路可推断正常的项
3. **本轮未覆盖项**:明确列出,不笼统表述"全部通过"

不允许用"完全闭环"等绝对表述,除非所有临时兼容已收口且所有链路已直接验证。

### D. 输出执行摘要

严格按 [AGENTS.md](../../AGENTS.md) §8 模板输出。关键要点:

- **模板是最低要求,不可简化**
- 主字段必须全填,无相关内容时写"无"或"未涉及",不能空白
- **下一轮上下文建议** 字段必须填写,指导下一轮 session 启动时的读取范围(哪些文件需要 / 不需要再读)
- 复杂任务可在模板基础上扩展,扩展字段不覆盖主字段

### E. 遗留问题收口顺序

验收后如发现已知遗留:

1. **优先先做收口包,再进入下一大包** — 不得叠加新功能同时收口
2. 收口包只做:
   - 去临时兼容
   - 补联调闭环
   - 补验收口径
   - 清理规则
3. 收口包不顺手叠加无关新功能

示例:执行包 A 验收后发现临时兼容未移除 → 先开收口包移除 → 再进入执行包 B。

### F. 文档与执行分层

- 文档级任务清单用于审查与对齐,不直接投喂执行工具
- Codex 执行必须以执行级任务清单为准
- 文档级与执行级不一致时 → 先修正文档,再执行代码任务

## 输出要求

完整的执行摘要,包含 [AGENTS.md](../../AGENTS.md) §8 模板所有主字段 + "下一轮上下文建议"。

若有遗留:

```text
### 遗留问题清单
- 已知未修复项:
  - 问题 1:[描述] [严重度] [建议收口时机]
- 是否需要收口包:是 / 否
- 收口包范围(若需):
```

## 完成判定

同时满足:

- 验收前置确认完成(路由 / 端口 / 鉴权一致)
- 最小验证链路执行,且验证类型与任务类型匹配
- 跨任务回归项已分类(直接 / 侧面 / 未覆盖)
- 执行摘要完整输出,主字段与"下一轮上下文建议"均填写
- 遗留问题已标记,收口顺序已明确

摘要未输出 → 任务未收口。遗留未标记 → 不得表述为"已完成"。
