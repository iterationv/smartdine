# AI 协作方法论

> 从 SmartDine 架构重构中沉淀的跨项目原则。
> 适用于任何需要多 AI 工具协作的项目(NTS / PL / 其他)。


## 1. 背景

2026 年 Q1,在 SmartDine 项目做了一次 AI 协作架构重构。
起因:CLAUDE.md 和 AGENTS.md 互相引用、内容重复,L0 启动消耗过大。
结果:L0 加载量下降 78-84%,流程细节下沉为可复用 Skill。
但真正的收获不是文件变小,而是对 AI 协作的底层认知升级。


## 2. 五个核心原则

### 2.1 工具入口分离

每个 AI 工具有自己的官方入口,不要强行统一:
- Claude Code → CLAUDE.md(官方约定自动读)
- Codex → AGENTS.md(官方约定自动读)
- 其他工具 → 查官方文档,顺势而为

**反例**:强制 Codex 先读 CLAUDE.md,每次指令都要额外说明,反向增加成本。

### 2.2 内容单向分层

三层结构,职责不重叠:
- 项目静态事实(三个月不变)→ CLAUDE.md 类入口
- 协作动态规则(阶段会变)→ AGENTS.md 类规则
- 专项流程(按场景触发)→ Skill 类手册

**判断标准**:三个月后这句话还会一字不变吗?
- 会 → 事实层
- 可能变 → 规则层
- 具体场景步骤 → 流程层

**反例**:把"禁止修改 X 文件"和"X 文件列表"写在同一个地方,未来调整时要改多处。

### 2.3 上下文治理

AI 编程的瓶颈不是模型能力,是上下文获取和管理能力。
核心机制:
- L0(自动读)只放启动必需,控制在单文件
- L1(按需读)声明理由后加载
- L2(触发读)有明确条件才加载
- 每轮任务产出执行摘要,作为下一轮 L0 压缩入口

**反例**:所有规则堆在一个文件,每次启动全量加载,80% 是当前任务用不到的。

### 2.4 模型分工

不是"哪个模型更强",是"哪个模型适合哪种视角":
- 规划视角(需求拆解 / Spec / 复盘)→ ChatGPT 5.5
- 执行视角(代码实现 / 修复)→ Codex(5.4 默认 / 5.5 兜底)
- 审查视角(交叉验证 / 找遗漏)→ Claude Code

**反例**:让同一个模型同时承担规划 + 执行 + 审查,角色混乱,自己审自己的方案找不出盲点。

### 2.5 方法论是基础设施

做到一定阶段后,你的竞争力不是"用了多少 AI 工具",是"你建设的 AI 协作系统"。
CLAUDE / AGENTS / Skill 不是提示词技巧,是项目级基础设施。


## 3. 五个落地技巧

### 3.1 三方视角收敛

高风险架构决策时,让 3 个视角依次审查,而不是单一模型一把梭:
- 一个出初稿(通常 GPT5.5)
- 一个推方案(通常 Claude)
- 一个挑漏洞(通常 Codex,有官方文档信息优势)

本次重构就因此抓到了"CLAUDE 作为统一入口"的基础事实错误。

**不适用于**:日常 CRUD 任务、简单修复、明确的执行型工作。
**适用于**:架构决策、跨模块重构、历史踩坑、需求歧义大的任务。

### 3.2 分阶段暂停点

长流程任务(5 步以上),在 AI 指令里强制设置暂停点,每步审阅后再放行。
本次 6 个暂停点抓到了 4 处需要调整的问题,如果一把梭,这些问题要到最后整体审才能发现。

### 3.3 行数上限是指导,不是法律

给 AI 设定文件行数目标时,留 10-15% 弹性。
如果卡字数会牺牲结构清晰度(目录树、接口契约、关键命令),**结构优先**。

### 3.4 Skill 顶部标注上位规则

每个 Skill 文件开头标注"上位规则:AGENTS §X / §Y"。
防止 Skill 演变成独立规则源,保持单一信息源。

### 3.5 阶段边界持续更新

AGENTS 中的"当前阶段"章节必须有"进入下一阶段时需检查本节是否过期"的自警提示。
否则阶段推进后,旧阶段的约束会变成僵尸规则,误导后续开发。


## 4. 什么时候用 / 不用这套方法论

**值得投入建设**:
- 项目跨越多个阶段(V1 → V2 → V3)
- 涉及多端 / 多模块协作
- 有多个 AI 工具参与开发
- 长期维护(3 个月以上)

**不值得投入建设**:
- 一次性脚本 / 原型验证
- 单人单端的小项目
- 仅用一个 AI 工具的工作流
- 周期短于 2 周的任务

强行在小项目上套这套方法论 = 过度工程化。


## 5. 迁移到新项目的最小清单

在新项目里复用这套方法论时,最少要做的 5 件事:

1. 建立工具入口文件(CLAUDE.md / AGENTS.md 或对应文件)
2. 明确三层内容归属(事实 / 规则 / 流程)
3. 定义 L0 / L1 / L2 读取规则
4. 定义模型分工(谁规划 / 谁执行 / 谁审查)
5. 至少建立 1 个 Skill(推荐先建 skill-task-intake)

其他(多 Skill、Git 规则、阶段边界)按需扩展,不求一步到位。


## 6. 持续演进

本文档本身也遵循"阶段边界持续更新"原则:
- 每完成一个项目的 AI 协作建设,回来更新一次
- 发现新原则补充到 §2,新技巧补充到 §3
- 过时的原则加注"已废弃"标记,不直接删除(保留演进历史)
