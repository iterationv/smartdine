## 一、P1 阶段目标

P1 的核心目标不是扩张业务范围，而是在 P0 已形成的产品骨架之上，补齐三类能力：

1. **增强自然语言能力**：提升对真实用户问法的适应性，减少“问得到但命不准”的问题。
2. **增强运营闭环**：让后台从“能看”升级到“能分析、能回流、能处理”。
3. **增强前台场景感**：让 H5 从基础问答页进一步接近真实服务入口。

一句话定义：

> 让 SmartDine 从“能稳定跑通的问答系统”，升级为“具备基础检索增强、运营闭环和场景化入口的产品雏形”。

---

## 二、P1 交付边界

### 2.1 本阶段要做

- 检索增强：Query Rewrite、BM25、Rerank、向量检索接口预留
- Admin 看板增强：趋势图、未命中处理状态、未命中转知识条目
- H5 场景入口增强：推荐问题接口化、快捷分类、相关推荐、结果页行动入口预留
- 工程治理：运行时日志文件移出 git tracking

### 2.2 本阶段不做

- 真正的向量检索实现
- 多轮对话上下文记忆
- 个性化推荐 / 营养分析 / 菜单联动
- 主后端技术栈迁移
- 大规模 Prompt 平台化

```text
执行包 A（纯后端）          ← 先执行
执行包 B（Admin）           ← 依赖 A 完成后启动
```

- **执行包 C** 主要解决前台入口与用户承接问题。
- 任务编号延续 P0，P1 从 **TASK-13** 开始。

## TASK-13｜Query Rewrite（查询改写）

- **编号：** TASK-13
- **标题：** Query Rewrite（查询改写）
- **执行包：** A
- **范围：**
  - 在原始问题未命中时，对问题做一次轻量改写后再次匹配。
  - 改写方向聚焦于去除冗余表达、保留核心意图，不引入复杂语义理解。
  - 改写结果仅作为二次检索输入，不改变用户原始提问内容。
- **验收标准：**
  - 常见口语化问题（如“你们有什么推荐菜？”）在改写后可形成更聚焦的查询表达。
  - 原始问题直接命中时，不触发改写流程。
  - 改写只影响检索过程，不改变 `/chat` 接口返回结构。
  - 问题日志中可识别是否触发过改写，用于后续观察效果。
- **依赖关系：**
  - 前置依赖：无
  - 后置依赖：为 TASK-14 提供更稳定的输入查询

---

## TASK-14｜BM25 混合检索

- **编号：** TASK-14
- **标题：** BM25 混合检索
- **执行包：** A
- **范围：**
  - 将当前简单交集计数式匹配升级为更合理的基于词频与文档长度归一化的检索评分方式。
  - 保留 P0 中已有的基础清洗与 alias 能力，不做全盘重写。
  - 目标是解决“高频虚词与核心关键词同权”导致的误命中问题。
- **验收标准：**
  - 对“推荐菜”类问题，推荐类知识条目排名应高于泛菜品类条目。
  - 对 P0 既有 20 条测试问题，命中率不能低于 P0 验收基线。
  - 检索结果排序较 P0 更稳定，尤其在模糊表达场景下改善明显。
  - 新算法引入后，不破坏知识条目现有字段和接口契约。
- **依赖关系：**
  - 前置依赖：TASK-13
  - 后置依赖：为 TASK-15 提供基础得分

## TASK-15｜Rerank（重排序）

- **编号：** TASK-15
- **标题：** Rerank（重排序）
- **执行包：** A
- **范围：**
  - 在检索阶段得到候选结果后，增加一层轻量重排序。
  - 综合 BM25 得分、alias 命中、tag 关联等信号，输出最终最优匹配项。
  - 目标是解决“候选都相关，但最优结果不够准”的问题。
- **验收标准：**
  - 同分或近似分候选中，语义更贴近用户问题的条目排在前面。
  - “推荐菜”类问题最终返回推荐类知识，而不是更泛的菜品类知识。
  - Rerank 仅增强排序，不改变 `/chat` 外部响应结构。
  - 经过 Query Rewrite + BM25 + Rerank 后，整体检索效果优于 P0。
- **依赖关系：**
  - 前置依赖：TASK-14
  - 后置依赖：无

# 九、风险与控制点

## 9.1 风险

- 检索增强后，可能提升部分问题准确率，但也可能导致部分旧问题回退。
- Admin 看板和状态闭环一旦字段设计不清，容易反复返工。
- H5 入口增强若过度发散，容易提前进入 P2 范围。

## 9.2 控制点

- 所有 P1 任务都必须以“不破坏 P0 已验收能力”为前提。
- 文档级任务清单只定义目标、范围、验收和依赖，不提前固化过细实现。
- 真正执行前，再拆为 Codex 可直接执行的任务清单。
