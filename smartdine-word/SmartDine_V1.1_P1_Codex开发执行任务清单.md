## 一、P1 总目标

在 P0 已完成基础问答闭环的前提下，进入 P1 产品骨架完善阶段，完成以下三类增强：

1. **后端检索增强**：提升自然语言问法下的命中准确率与排序质量。
2. **Admin 运营闭环**：将未命中问题转化为可运营、可沉淀、可追踪的知识建设入口。
3. **H5 场景优化**：让用户端从"能问能答"升级为"更易问、更易看、更易继续操作"。

## TASK-A1：Query Rewrite（未命中改写重试）

### 目标

在第一次知识匹配未命中时，对用户原始问题做一次轻量改写，再尝试第二次匹配；若仍未命中，再进入 AI fallback。

### 范围

- 改写仅限轻量规则，不引入外部模型
- 只处理高频口语冗词、礼貌词、主语包装词
- 不允许把原问题语义改写到偏离原意

### 必须涉及文件

- `smartdine-api/src/ai/retrieve.ts`（插入 rewrite → 二次匹配控制流）
- `smartdine-api/src/ai/` 下新增改写模块（文件名由执行者按现有命名风格确定，如 `queryRewrite.ts`）
- `smartdine-api/src/types/` 下按需补充类型

> ⚠️ TASK-A1 期间不重构 `matchKnowledge.ts` 内部评分逻辑，该工作归属 TASK-A2。

### 具体要求

1. 保留原始 `question` 作为第一轮匹配输入。
2. 第一轮 `matchKnowledge(question)` 未命中时，再对 `question` 执行 rewrite。
3. 若 rewrite 结果与原问题等价或为空，则不做第二轮，直接进 fallback。
4. 第二轮使用 rewrite 后的问题再次匹配。
5. 若第二轮命中，返回结果中应能保留"原问题"和"改写后问题"的可追踪信息，供后续日志或调试使用。
6. 未命中时仍走原有 AI fallback，不改变 `/chat` 的基础响应契约。

### rewrite 处理方向（规则型，不做 LLM 改写）

- 去礼貌词：如"请问""你好""麻烦问下"
- 去包装词：如"你们""这里""食堂里""有没有""有什么"
- 保留核心关键词：如"推荐菜""退款规则""营业时间"

### 验收标准

- "你们有什么推荐菜？"这类问法，在不改知识数据的前提下，命中准确率提升
- rewrite 不影响原本可直接命中的问题（第一轮已命中时不触发改写）
- `/chat` 基础响应结构不被破坏
- 日志或调试信息中可区分 original query 与 rewritten query

### 依赖关系

- 无前置依赖
- 为 TASK-A2 / TASK-A3 提供更稳定的输入基础

## TASK-A2：BM25 混合检索并入现有 retrieve 链路

### 目标

将现有偏简单的 bigram 交集命中逻辑，升级为更适合问答检索的小规模 BM25 评分能力，并并入当前 retrieve / match 链路。

### 范围

- 以现有知识条目为语料库
- 允许继续复用现有 normalize/tokenize 思路
- 不引入数据库，不引入外部搜索服务

### 必须涉及文件

- `smartdine-api/src/ai/matchKnowledge.ts`（重构内部评分核心，拆分候选召回与最终返回）
- `smartdine-api/src/ai/` 下新增 BM25 评分模块
- `smartdine-api/src/data/knowledgeStore.ts`
- 按需补充 `types`

### 具体要求

1. 保留现有知识加载方式，不改数据源位置。
2. 以知识标题、question、aliases、tags 等可检索文本为语料基础。
3. **将"候选召回 + 分数计算"与"最终返回最佳候选"分开组织**，为 TASK-A3 的 rerank 留出明确插入点。
4. BM25 结果至少输出：
   - 候选条目 id
   - 原始检索分数
   - 命中的文本证据或命中来源（如主问题 / 别名 / 标签）
5. 不要求一次性删除所有旧逻辑，但最终主流程必须以 BM25 结果为核心输入。
6. 兼容小规模语料场景，不追求搜索系统级别复杂度。

### 验收标准

- 推荐类问法、近义问法、口语问法的基础召回更稳定
- 相比原有简单交集计数，候选排序更符合语义直觉
- 代码结构上已为 rerank 留出明确插入点（有注释或结构体现）

### 依赖关系

- 前置依赖：TASK-A1
- 为 TASK-A3 提供候选集与基础分数

## TASK-A3：Rerank（对检索结果重排序）

### 目标

针对 BM25 候选集进行二次排序，解决"高频虚词与业务关键词同权""同分时靠顺序取胜"等问题。

### 范围

- 仅针对检索候选集重排，不引入外部 rerank 模型
- 允许使用规则加权：alias 命中加权、tag 命中加权、question 精确度加权

### 必须涉及文件

- `smartdine-api/src/ai/` 下新增 rerank 模块（通过 A2 留出的插入点接入，不整体改写 `matchKnowledge.ts`）
- 按需补充类型

### 具体要求

1. 将 BM25 召回出的前 N 个候选输入 rerank。
2. rerank 至少考虑：
   - alias 是否直接命中
   - question 是否更接近用户核心表达
   - tags 是否命中业务关键词
   - rewrite 前后输入的一致性参考
3. **所有加权系数集中在 rerank 模块内定义**，不允许魔法数字分散在多个文件中。
4. 最终最佳条目应基于 rerank 后结果返回。
5. 对关键误判样例（如"你们有什么推荐菜？"）做定向验证。

### 验收标准

- "推荐菜"相关问题优先命中推荐类知识条目，而不是因为"有什么"这类高频词误命中普通菜品条目
- 同分时不再简单按知识文件出现顺序决定结果
- rerank 逻辑可解释，可通过日志或调试信息查看重排依据

### 依赖关系

- 前置依赖：TASK-A2
- 可利用 TASK-A1 的 rewrite 信息提升排序质量

## TASK-B3：missedQuestions 增加 handled 字段

### 目标

为未命中问题增加"是否已处理"状态，支持 Admin 侧区分待处理与已处理问题。

**语义说明**：`handled` 表示运营已对该问题做出响应，不等同于"已转知识条目"。以下情形均可标记为 handled：
- 已转为知识条目（由 TASK-B2 自动触发）
- 人工查看后确认忽略
- 标记为暂不处理

### 范围

- 数据层新增字段
- 接口返回补齐字段
- Admin 列表支持展示、筛选与更新

### 必须涉及文件

- `smartdine-api/src/types/log.ts`
- `smartdine-api/src/data/logStore.ts`
- `smartdine-api/src/routes/logs.ts`
- `smartdine-api/src/services/logService.ts`
- `smartdine-admin/src/api/logs.js`
- `smartdine-admin/src/views/.../MissedList*.vue`

### 具体要求

1. 为 missed question 数据结构增加 `handled` 字段（boolean，默认 `false`）。
2. 历史数据需兼容未带该字段的情况（读取时缺省视为 `false`）。
3. Admin 列表支持展示 handled 状态。
4. Admin 提供最小可用的处理动作：手动标记已处理 / 撤销处理。
5. **Admin 支持按处理状态筛选**（待处理 / 已处理）。
6. 与"转知识条目"动作配合时，自动将 handled 置为 `true`。

### 验收标准

- 未命中问题可区分待处理与已处理
- 支持按处理状态筛选
- 转知识条目后 handled 状态自动变为已处理
- 历史数据不因字段新增报错
- 状态持久化，刷新后保留

### 依赖关系

- 前置依赖：TASK-A5 完成
- 建议与 TASK-B2 同包收口，共用 handled 字段

## TASK-C1：推荐问题接口化

### 目标

将当前 H5 推荐问题从本地写死或静态配置，升级为后端接口下发，便于后续运营调整与动态优化。

### 范围

- 新增推荐问题接口
- H5 通过接口获取推荐问题
- 保留最小回退方案，避免接口失败时页面空白

### 必须涉及文件

- `smartdine-api` 下新增 suggestions 接口（按现有 routes 风格组织）
- `smartdine-h5/src/api/` 下对应接口调用文件
- `smartdine-h5/src/stores/` 下推荐问题相关 store（如已有则复用）
- `smartdine-h5/src/views/Chat.vue`

### 具体要求

1. 后端提供推荐问题接口，推荐内容可基于现有知识条目与日志热度动态生成（最小实现可先返回固定知识条目精选，不强制实现热度排序）。
2. H5 页面加载时拉取推荐问题。
3. 接口失败时使用前端硬编码兜底列表展示，不出现空白或报错。
4. **推荐问题数据结构需携带分类信息**（category 字段），便于 TASK-C2 的分类 Tab 筛选使用。
5. 推荐问题数据结构保持简单稳定，便于后续扩展排序。

### 验收标准

- H5 推荐问题由接口驱动，不再完全写死在前端
- 接口失败时页面仍可正常展示兜底推荐
- 推荐问题携带 category 字段，结构已为分类 Tab 准备就绪
- 不影响原有提问主流程

### 依赖关系

- 可与执行包 B 并行，建议在 A 完成后启动
- 为 TASK-C2 提供带分类信息的推荐数据来源

## TASK-C2：快捷分类 Tab

### 目标

为 H5 增加快捷分类入口，让用户不必完全依赖自由输入，可通过分类快速发现高频问题。

### 范围

- 分类入口展示
- 分类与推荐问题联动
- 不做复杂二级分类系统

### 必须涉及文件

- `smartdine-h5/src/views/Chat.vue`
- `smartdine-h5/src/components/` 下按需新增分类 Tab 组件
- `smartdine-h5/src/stores/` 下推荐问题 store

### 具体要求

1. 至少提供一组一级分类 Tab，例如：推荐、菜品、营业时间、规则等，分类以现有知识标签体系为基础。
2. 切换分类后，推荐问题列表随之过滤变化。
3. **无推荐数据时有明确空态提示**，不出现视觉空白。
4. UI 保持 H5 轻量风格。

### 验收标准

- 用户可通过 Tab 直接浏览不同类别推荐问题
- Tab 切换与推荐问题联动正常
- 空态有明确提示
- 移动端样式正常

### 依赖关系

- 前置依赖：TASK-C1（需推荐数据携带 category 字段）

## TASK-C3：相关推荐 + 结果页 CTA 预留

> **合并说明**：相关推荐（原 C3）与结果页 CTA 预留（原 C4）涉及完全相同的文件（`Chat.vue` / `MessageList.vue`）和相同的结果区结构，拆分执行会导致同一区域被改动两次。合并为一个 TASK 统一收口，减少重复改动和交叉风险。

### 目标

在问答结果页中增加相关推荐问题（提升连续提问效率），同时预留结果页 CTA 入口（为 P2 业务跳转占位）。

### 范围

- `/chat` 响应可选扩展 `related` 字段
- H5 在回答结果区域展示相关推荐
- 结果区预留 CTA 占位按钮
- 不破坏现有 `/chat` 响应兼容性
- CTA 当前只做占位交互，不接真实业务路由

### 必须涉及文件

- `smartdine-api/src/routes/` 与 `retrieve / match` 相关链路（`related` 字段扩展）
- `smartdine-h5/src/views/Chat.vue`
- `smartdine-h5/src/components/MessageList.vue`

### 具体要求

**相关推荐部分：**
1. 后端根据命中条目的 tags、分类或相近知识条目，在响应中可选返回 `related` 字段（推荐问题列表）。
2. `related` 作为可选字段，旧逻辑不依赖该字段也能正常工作。
3. **未命中或 AI fallback 场景下，不强行展示相关推荐**（`related` 返回空或不返回）。
4. H5 在回答区域展示相关推荐，并支持点击二次提问。

**结果页 CTA 预留部分：**
5. 根据回答结果或标签，展示"查看详情""去看看""进入功能页"等 CTA 预留按钮。
6. 当前点击后仅 Toast 提示"即将上线"，不做真实跳转。
7. **AI fallback 类回答不展示业务 CTA 入口**（避免无意义引导）。
8. 组件结构需为 P2 真正接入业务页保留清晰位置。

### 验收标准

- 命中知识条目后可看到相关推荐，点击可触发新一轮提问
- 未命中 / fallback 场景下不展示相关推荐和 CTA
- 结果页可展示 CTA 预留入口，点击有明确 Toast 反馈，不出现死按钮
- 不破坏原有消息渲染逻辑
- 后续接入真实业务时无需大改结果页结构

### 依赖关系

- 前置依赖：TASK-C1 / TASK-C2
- 可部分复用执行包 A 的 tags / 检索信息
