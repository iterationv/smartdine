## 一、P2 总目标

P2 的目标不是继续堆功能，而是在 P1 已经跑通的 FAQ + API + Admin + H5 基础上，继续补齐真实使用前必须收口的三类问题：

1. **问答入口更可信**
   - 推荐问题要从"全部知识条目展示"升级为"有策略的推荐"。
   - 检索链路要减少"局部关键词命中导致误答"的情况。
   - Query Rewrite 不强行证明收益，等待 KB 扩展或语义检索条件成熟后再验证。

2. **Admin 更接近可运营后台**
   - 临时登录方案要退出主流程。
   - AI 模型、Prompt、回答口径、兜底话术等配置要有清晰归属。
   - 关键问答事件需可追溯，为后续运营决策提供基础数据。
   - P2 只做最小正式化，不做完整权限系统和 Prompt 平台。

3. **工程状态继续变干净**
   - 构建产物、测试产物不应持续污染 git diff。
   - H5/API 分类口径要有同步约束。
   - Admin bundle warning 后置处理，不抢主链路优先级。

## 三、执行前统一约束

### 3.1 必须遵守

1. 不修改 `.env` / `.env.ai` / `.env.cc` 中的真实密钥。
2. 不引入 RAG、向量库、embedding 实装。
3. 不引入多轮对话上下文记忆。
4. 不引入多账号、多角色、多租户权限体系。
5. 不做 Admin + H5 整体视觉重构。
6. 不做大规模目录重组。
7. 不重开 P1 已验收范围。
8. 所有任务必须可单独验收。
9. 每个执行包完成后必须输出验收结果。
10. 发现回归问题走回归修复流程，不混入 P2 功能任务。

### 3.2 推荐工作流

```text
文档级任务确认
→ 生成 Codex / Claude Code 执行清单
→ 单包执行
→ 单包验收
→ 交叉校验
→ 归档问题
→ 决定是否进入下一包
```

### 3.3 范围冻结

本清单一旦确认，进入执行阶段后的任何新增需求按以下流程处理：

1. **不抢当前执行包**：当前执行包按已冻结范围继续，不中途追加。
2. **登记变更**：在 `SmartDine_V1.1_P2_变更登记.md` 中记录新增需求来源、必要性、影响范围。
3. **分类处理**：
   - 属于已有任务遗漏的细节 → 当前任务执行内补齐，验收时一并检查。
   - 属于新功能 → 评估是否进入 P2 后续执行包，或推迟到 P3。
   - 属于回归问题 → 走 P1 回归修复流程，单独 commit，不混入 P2 功能任务。
4. **避免范围漂移**：若 P2 范围在执行中扩张超过 30%，触发整体重新收敛，不强行推进。

## TASK-P2-01｜工程清理：构建产物与测试产物移出 git tracking

### 对应来源

P2-05

### 目标

延续 P1 工程清理思路，把仍被 git tracking 的构建产物和测试产物移出版本管理，降低后续每次构建、测试造成的脏 diff。

### 范围

重点检查并处理：

- `smartdine-api/dist/`
- `smartdine-admin/test-results/.last-run.json`

允许同步检查：

- `smartdine-admin/dist/`
- `smartdine-h5/dist/`
- `coverage/`
- `playwright-report/`
- 其他明显运行期或测试期产物

### 不做

- 不删除本地构建能力。
- 不删除测试报告功能。
- 不改业务代码。
- 不改真实环境变量。
- 不顺手重构目录结构。

### 建议动作

1. 只读检查当前 git tracking 状态。
2. 确认哪些产物已被纳入 git。
3. 使用 `git rm --cached` 移出 tracking。
4. 更新 `.gitignore`。
5. 同步更新 Git 规范文档中的工程清理说明。
6. 构建或测试一次，确认不会重新产生脏 diff。
7. 输出清理报告。

### 验收标准

- 构建后 `smartdine-api/dist/` 不再出现在 git diff 中。
- Playwright 或测试运行后 `.last-run.json` 不再污染 git status。
- `.gitignore` 与实际产物一致。
- Git 规范文档不再保留错误的过渡性描述。
- 没有误删源码文件。
- 没有把真实日志、真实密钥或测试缓存提交。

### 风险点

- `git rm --cached` 路径写错可能误移出源码。
- `.gitignore` 规则过宽可能误忽略应提交文件。
- 本地已有产物与仓库历史不一致，容易误判为删除。

### 建议提交类型

```text
chore(git): remove build and test artifacts from tracking
```

# 五、执行包 A：推荐与检索可信度

> 执行包 A 内部顺序：先 TASK-P2-04 建立样例集，再并行 TASK-P2-02 / TASK-P2-03，最后由 TASK-P2-11 接入可观测性。

## TASK-P2-04｜推荐与检索回归样例集整理（前置）

### 目标

为 P2 推荐策略和低置信度兜底建立最小回归样例集，**作为 TASK-P2-02 / TASK-P2-03 的验收依据**，避免后续优化导致旧问题回退。

### 为什么前置

- TASK-P2-02 / P2-03 的验收都需要"可重复执行的样例清单"。
- 样例集先于实现存在，能避免开发者依据自己的实现回头编造样例。
- 样例集本身工作量小，由 Claude Code 整理后可立即被 Codex 执行任务消费。

### 范围

整理三类样例：

1. 正常高频问题。
2. 模糊口语问题。
3. 极端复合/未知条件问题。

样例需要覆盖：

- 推荐菜。
- 支付方式。
- 营业时间。
- 菜品价格。
- 充值/退款。
- 外卖/打包。
- 未知问题。
- 多意图混合问题。

### 不做

- 不追求覆盖全部业务。
- 不引入完整自动化测试平台。
- 不造假 Query Rewrite 收益。
- 不把样例写死成唯一真理，后续可扩展。

### 验收标准

- 至少形成一份可重复执行的样例清单（建议 ≥ 30 条，覆盖三类）。
- 每条样例有预期结果。
- P2-02、P2-03 执行后可用该清单验收。
- 能区分"应命中""应低置信度""应 fallback"三种类型。
- 样例集归档到文档或测试脚本中。

### 建议样例结构

```json
{
  "question": "你们食堂支不支持火星币支付和月球外卖？",
  "expectType": "low_confidence",
  "expectNotContain": ["支持火星币", "支持月球外卖"],
  "note": "未知附加条件不应被局部支付意图覆盖"
}
```

### 依赖关系

- 必须在 TASK-P2-02 / TASK-P2-03 启动前完成初版。
- 后续作为 P2 总体验收输入。

### 建议提交类型

```text
test(api): add retrieval regression cases
docs(p2): add retrieval acceptance cases
```

## TASK-P2-02｜推荐问题策略升级

### 对应来源

P2-01

### 目标

将 `GET /api/suggestions` 从"返回全部 active knowledge"升级为"可控、有限、有排序策略的推荐问题接口"。

### 业务问题

当前知识量少时，返回全部 active knowledge 还能接受；知识量增长后，首页推荐区会退化为"全部问题列表"，失去推荐意义。

### 范围

后端建议支持：

- 推荐数量限制。
- active 条目过滤。
- 分类均衡。
- 基础排序策略。
- 默认推荐兜底。
- 可解释的推荐来源字段。

H5 建议支持：

- 接口返回数量变化。
- 分类切换时保持展示稳定。
- 接口异常时保留 fallback。
- 推荐为空时展示合理空态。

### 推荐策略分层

#### 第一层：最小可落地策略（P2 必做）

- 只返回前 N 条（建议默认 N=8，可通过 query 参数或环境变量调整）。
- 仅返回 active 条目。
- 优先使用显式推荐字段或排序字段。
- 没有排序字段时使用稳定默认排序（按 id 或 createdAt）。

#### 第二层：分类均衡策略（P2 必做）

- 每个主要分类最多展示固定数量（建议每类 ≤ 3）。
- 避免某个分类占满首页。
- tags 为空的条目进入兜底分类（与 TASK-P2-08 保持一致）。

#### 第三层：热度策略（P2 不做，仅预留接口）

- 基于日志统计点击或提问次数。
- 按热度排序。
- 当前 P2 可以预留字段位，不强制上线。

### 不做

- 不做个性化推荐。
- 不做用户画像。
- 不做复杂推荐算法。
- 不做营养分析联动。
- 不做菜单联动。
- 不引入新的数据库。

### 验收标准

- API 不再无脑返回全部 active knowledge。
- 推荐数量可控，默认上限明确（建议 8 条）。
- 知识条目增加后，首页推荐仍保持合理数量。
- H5 首页展示稳定，不出现空白。
- fallback 逻辑仍有效。
- 老接口调用方不被破坏（保留响应字段兼容）。
- 推荐结果具备基础可解释性（返回来源字段，如 `sourceType: "active" | "fallback"`）。
- 通过 TASK-P2-04 样例集中"推荐类"样例。

### 建议验证样例

```text
样例 1：知识库 active 条目少于推荐上限
期望：全部 active 条目可返回，顺序稳定。

样例 2：知识库 active 条目多于推荐上限
期望：只返回限定数量，不撑爆首页。

样例 3：多个分类都有知识条目
期望：推荐区不被单一分类占满。

样例 4：接口异常
期望：H5 使用 fallback，不空屏。

样例 5：tags 为空
期望：归入明确兜底分类，不导致前端异常。
```

### 依赖关系

- 依赖 P1 推荐问题接口化能力。
- 依赖 TASK-P2-04 样例集已就绪。
- 可与 TASK-P2-03 并行。
- 不依赖 Admin 认证。

### 建议提交类型

```text
feat(api): improve suggestion ranking and limit strategy
feat(h5): adapt suggestion display to ranked suggestions
```

## TASK-P2-03｜检索低置信度与复合问题兜底

### 对应来源

P2-03

### 目标

减少极端复合问题、未知条件混入问题导致的误答，让系统在不确定时能提示、澄清或降级，而不是过度自信地返回局部命中答案。

### 业务问题

例如：

```text
你们食堂支不支持火星币支付和月球外卖？
```

这类问题中包含部分已知意图"支付"，但也包含明显未知条件"火星币""月球外卖"。如果系统直接命中"支持支付方式"，会显得答非所问。

### 低置信度判定信号（必须在实现中明确）

实现时**必须明确以下三类信号的判定阈值**，并在代码中以可配置常量存在（不写魔法数字）：

#### 信号 1：top1 绝对分数过低

- 当 BM25 top1 score 低于绝对阈值 `LOW_CONFIDENCE_ABS_THRESHOLD` 时，判定为低置信度。
- 阈值初始建议值需在 TASK-P2-04 样例集上回归后确定，不在本文档中预先固定数值。

#### 信号 2：top1 与 top2 分差过小

- 当 `(top1.score - top2.score) / top1.score < LOW_CONFIDENCE_GAP_RATIO` 时，判定为多候选模糊。
- 此时进入策略 B（多候选展示），而不是直接返回 top1。

#### 信号 3：未知实体占比过高

- 对 query 做最简分词后，若有 ≥ N 个连续 token 在所有候选文档中均无任何匹配（包括 tags、问题、答案），判定为含未知附加条件。
- 此时进入策略 C（未知条件兜底）。
- 不引入 NER，仅基于现有 BM25 词项命中信息即可实现。

#### 信号优先级

```text
信号 3（未知实体）> 信号 1（绝对分低）> 信号 2（候选模糊）
```

即：未知实体占比高时，直接走策略 C，不再进入策略 A/B 判断。

### 范围

后端建议补充：

- 上述三类信号的判定函数。
- 阈值常量集中管理（建议放在 `src/retrieval/confidence.ts` 或同等位置）。
- 低置信度返回字段（建议在响应中加 `confidence: "high" | "low" | "ambiguous" | "unknown_entity"` 与 `fallbackReason`）。
- 兜底话术配置（话术文本可被 TASK-P2-06 后续接管，本任务先以默认常量实现）。

H5 建议补充：

- 低置信度提示展示。
- 引导用户换一种问法。
- 多候选时展示候选列表，让用户点击。
- 不把低置信度回答伪装成确定答案（视觉上可加"我不太确定"前缀）。

### 三种回答策略

#### 策略 A：低置信度提示（信号 1 触发）

```text
我不太确定你问的是不是"支付方式"。你可以换一种问法，或者点击下面的推荐问题。
```

#### 策略 B：多候选展示（信号 2 触发）

```text
你可能想问：
1. 支持哪些支付方式？
2. 是否支持外卖？
```

#### 策略 C：未知附加条件兜底（信号 3 触发）

```text
目前知识库中没有找到"火星币"或"月球外卖"的相关说明。关于常规支付方式，我可以为你说明。
```

### 不做

- 不引入复杂 NLP 实体识别服务。
- 不接大模型做每次置信度裁判。
- 不做多轮澄清对话。
- 不做向量检索实装。
- 不改变 `/chat` 的基础兼容结构（仅扩展字段，老调用方不受影响）。

### 验收标准

- 三类判定信号在代码中以命名常量存在，不存在魔法数字。
- 极端复合问题不再直接返回确定性错误答案。
- 常规问题命中不受明显影响（TASK-P2-04 样例集中"应命中"类样例通过率 ≥ 95%）。
- "应低置信度"类样例触发预期信号的比例 ≥ 80%。
- "应 fallback"类样例不输出任何业务确定性答案。
- 响应中可观测到 `confidence` 与 `fallbackReason` 字段。
- H5 能展示对应策略的提示文案。
- P1 已验证的高频问题不大面积回退（回归样例通过率 ≥ 90%）。

### 建议验证样例

```text
样例 1：你们支持微信支付吗？
期望：confidence=high，正常命中支付方式。

样例 2：你们食堂支不支持火星币支付和月球外卖？
期望：confidence=unknown_entity，触发策略 C。

样例 3：有什么推荐菜？
期望：confidence=high，正常命中。

样例 4：我想知道早餐和充值还有外卖
期望：confidence=ambiguous，触发策略 B。

样例 5：abcxxx月球火星支付
```

## TASK-P2-05｜Admin 真实认证最小化改造

### 对应来源

P2-04

### 目标

将 Admin 从临时硬编码登录方案升级为最小真实认证方案，使其适合演示、部署和后续运营配置承载。

### 业务问题

当前 `admin / admin123` 硬编码 + localStorage 更适合早期调试，不适合长期存在。P2 需要让 Admin 至少具备"可配置、可替换、可说明"的认证边界。

### 拍定方案（不再保留多选）

**P2 采用方案：环境变量 + JWT。**

- 账号密码来源：服务端读取 `ADMIN_USERNAME` 与 `ADMIN_PASSWORD_HASH`（bcrypt 哈希，不存明文）。
- 登录成功后，服务端使用 `ADMIN_JWT_SECRET` 签发 JWT，有效期建议 24 小时。
- Admin 前端在登录后将 JWT 存入 `httpOnly` cookie 或 sessionStorage（择一，由实现者根据现有架构决定，但**不得使用 localStorage 存 JWT**）。
- 受保护接口在中间件层校验 JWT，校验失败返回 401。
- 退出登录由前端清除存储 + 服务端可选维护短期黑名单（最小实现可不维护黑名单，依赖 token 自然过期）。

**为什么这样选**：

- 排除"环境变量化账号密码（无 token）"：等于硬编码搬家，不解决"未登录也能访问受保护接口"问题。
- 排除"完整 refresh token 体系"：超出 P2 范围，token 过期重新登录即可。
- bcrypt + JWT 是行业最小可信方案，工作量约 1 天。

### 范围

后端：

- 新增 `/api/admin/login` 端点，接收用户名密码，校验后签发 JWT。
- 新增 `/api/admin/logout` 端点（最小实现可仅返回成功，由前端清存储）。
- 新增 JWT 校验中间件，应用于现有 Admin 受保护路由。
- 环境变量模板更新：`ADMIN_USERNAME`、`ADMIN_PASSWORD_HASH`、`ADMIN_JWT_SECRET`。
- 提供生成 `ADMIN_PASSWORD_HASH` 的辅助脚本（一次性使用）。

Admin 前端：

- 登录页对接新端点。
- 移除硬编码 `admin / admin123` 与"临时登录"提示。
- 请求拦截器统一附带 JWT。
- 401 响应统一跳转登录页。
- 退出按钮清理本地 token。

### 不做

- 不做用户注册。
- 不做多账号。
- 不做角色权限。
- 不做多租户。
- 不做密码找回。
- 不做验证码。
- 不做 OAuth / SSO。
- 不做 refresh token。
- 不做密码强度校验（账号由部署方设置，不开放注册）。

### 验收标准

- Admin 不再依赖硬编码 `admin / admin123`。
- 账号密码由环境变量配置，密码以 bcrypt 哈希存储。
- 未登录访问 Admin 受保护接口返回 401。
- 登录后可正常访问现有 Admin 功能。
- 退出后再访问受保护接口返回 401。
- localStorage / sessionStorage / cookie 中不存储明文密码。
- JWT 过期后访问受保护接口返回 401，前端正确处理。
- UI 中不再显示"临时登录方案"类提示。
- 不影响 H5 问答链路。
- `.env.example` 已更新，包含三个新环境变量与说明。

### Secret 管理约定

SmartDine V1.1 当前不引入外部 secret 管理工具(Vault / AWS Secrets Manager / 1Password 等),P2 阶段采用最小可信约定:

#### 来源与存储

- 所有 secret(`ADMIN_JWT_SECRET`、`ADMIN_PASSWORD_HASH`、AI 相关 API Key 等)统一通过 `.env` / `.env.ai` / `.env.cc` 文件提供。
- `.env` 文件不进入 git(已在 `.gitignore` 中)。
- `.env.example` 只包含变量名与说明,**不包含真实值**。
- 真实的 `.env` 文件由部署人员在目标机器上手动维护,通过受限渠道(私聊、加密邮件、或部署平台环境变量配置)传递。

#### 生成规范

- `ADMIN_JWT_SECRET`:至少 32 字节随机字符串,推荐使用 `openssl rand -base64 32` 生成。
- `ADMIN_PASSWORD_HASH`:由 P2-05 提供的辅助脚本(基于 bcrypt,cost ≥ 10)生成,生成后清理脚本输入历史。
- 同一个 secret 不在多个环境(dev / staging / prod)间复用。

#### 轮换策略

- `ADMIN_JWT_SECRET`:建议每 90 天轮换一次,或在以下情况立即轮换:
  - 部署人员变动。
  - 怀疑 secret 泄露。
  - 任何 `.env` 文件意外提交到 git(即使后续 rebase 删除,也视为泄露)。
- `ADMIN_PASSWORD_HASH`:管理员密码同样建议 90 天轮换,且不与个人其他系统密码相同。
- 轮换操作记录到 `SmartDine_V1.1_部署日志.md`(若不存在则创建),只记录"何时由谁轮换",不记录具体值。

#### 应急流程

- 若 `ADMIN_JWT_SECRET` 泄露:立即在服务端替换 secret 并重启,所有已签发 token 自动失效,管理员重新登录。
- 若 `.env` 误提交到 git:立即轮换其中所有 secret,然后再处理 git 历史(`git filter-branch` 或新建仓库),不可只删 commit 不换 secret。
- 应急后必须复盘并更新本节内容。

#### 不做

- 不引入外部 secret 管理服务。
- 不做 secret 自动轮换。
- 不做 secret 多版本共存(轮换即覆盖)。
- 不在代码或日志中输出任何 secret 值,即使是部分截断。

### 风险点

- 认证改造容易影响现有 Admin API 调用,需要全量回归 Admin 功能。
- JWT secret 泄露后所有 token 失效,应急流程见上方"Secret 管理约定"。
- 如果同时改太多权限逻辑,会偏离 P2,需严格守住"只有一个管理员账号"的边界。
- `.env` 文件管理依赖部署人员的纪律,V1.1 阶段无技术强制手段,需通过文档与流程约束。

### 依赖关系

- 依赖 TASK-P2-06 方案完成（确认 AI 配置接口的认证形式与本任务一致）。
- 不依赖 TASK-P2-11。

### 建议提交类型

```text
feat(api): add minimal admin authentication with jwt
feat(admin): replace temporary login with jwt-based auth
chore(env): add admin auth environment variables
```

# 八、执行包 D：依赖触发型验证

## TASK-P2-10｜Query Rewrite 自然样例收益验证

### 对应来源

P2-06

### 目标

在 KB 扩展或向量检索条件成熟后，重新验证 Query Rewrite 是否对真实问题召回有实际贡献。

### 当前状态

**暂缓。**

当前 KB + BM25 阈值组合下，不存在稳定的"原始未命中 → 改写后命中"自然闭环。不能为了证明 Query Rewrite 有用而构造不真实样例。

### 触发条件

满足任一条件再启动：

1. KB 明显扩展，出现更多短语化、口语化、别名化入口。
2. 向量检索真正落地。
3. P2 检索回归样例集中出现 rewrite 可改善的真实问题。
4. P1/P2 日志（TASK-P2-11）中积累到足够多真实未命中样本。

### 范围

- 收集真实问题样例。
- 标记原始 query 命中情况。
- 标记 rewrite 后命中情况。
- 对比是否提升召回。
- 分析是否造成误命中。
- 输出验证报告。

### 不做

- 不接入向量库。
- 不强行修改 rewrite 策略。
- 不伪造收益。
- 不影响当前 `/chat` 主链路。

### 验收标准

- 验证样例来自真实日志或明确扩展后的 KB。
- 报告能说明 rewrite 的收益、无收益或副作用。
- 如果收益不足，明确建议保持当前策略或降低 rewrite 权重。
- 如果收益明显，再进入后续执行任务。

### 建议输出

```text
SmartDine_V1.1_P2_QueryRewrite收益验证报告.md
```

### 建议提交类型

```text
test(api): verify query rewrite recall contribution
```

## 9.7 性能与一致性验收

- H5 推荐分类 fallback 与 API 口径一致。
- tags 为空有明确处理。
- Admin bundle warning 有处理或记录。

# 十三、结论

P2 可以拆成一句话：

> 先清理工程噪音，再让推荐和检索变可信、可观测，然后让 Admin 承接认证与 AI 配置，最后补齐一致性和性能问题。

推荐落地节奏：

```text
P2 第一轮：工程清理 + 样例集 + 推荐策略 + 检索低置信度 + 可观测性 + AI 配置方案
P2 第二轮：Admin 认证 + AI 配置最小落地（条件触发）
P2 第三轮：分类口径同步 + bundle 拆包
P2 延后项：Query Rewrite 收益验证
```

该拆分能保持 P2 聚焦，不提前进入 P3 视觉重构，也不提前透支 RAG、向量检索、多轮对话、多租户权限等后续阶段能力。
