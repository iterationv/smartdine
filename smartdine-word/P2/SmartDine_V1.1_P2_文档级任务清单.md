## 一、P2 定位

P2 不是 P1 的返工，也不是 P3 视觉重构的提前启动。

P2 的定位是：在 P1 已完成“FAQ + API + Admin + H5 最小闭环”的基础上，继续补齐影响真实可用性的三类能力：

1. **问答入口更可信**
   - 推荐问题不能退化成“全部问题”。
   - 检索不能因为局部关键词命中而误答极端复合问题。
   - Query Rewrite 暂不强行证明收益，等 KB 或语义检索条件成熟后再验证。

2. **Admin 从临时工具走向可运营工具**
   - 登录方案不能长期停留在硬编码账号。
   - AI 模型、Prompt、回答口径等运行配置，不能长期写死在代码里。
   - 但 P2 只做“最小可管理化”，不做完整 AI 平台。

3. **工程状态继续变轻、变干净**
   - 构建产物、测试产物不应继续污染 git diff。
   - H5/API 分类兜底口径需要同步约束。
   - Admin 大包问题可以保留在 P2 后段，不抢核心链路优先级。

## 二、P2 主线收敛

### 主线 A：推荐与检索可信度

目标：让用户进入 H5 后看到的推荐问题更像“被运营过的问题”，让机器人在低置信度场景下少误答。

对应任务：

- P2-01：推荐问题策略升级
- P2-03：检索低置信度与复合问题兜底
- P2-06：Query Rewrite 自然样例验证，延后触发

### 主线 B：Admin 正式化与 AI 运营配置

目标：让 Admin 不再只是临时调试入口，而是逐步承担运营配置能力。

对应任务：

- P2-04：Admin 真实认证最小化改造
- P2-08：AI 模型与 Prompt 配置后台化方案设计

说明：

P2-08 不是完整 Prompt 平台，也不是多模型运营平台。它只解决一个问题：后续模型供应商、模型名、系统提示词、回答边界、兜底话术等内容，应该有明确的配置来源和管理口径，而不是散落在代码里。

### 主线 C：工程治理与一致性补强

目标：减少脏 diff、减少隐性口径不一致、降低后续验收噪音。

对应任务：

- P2-05：git tracking 工程清理
- P2-07：H5 推荐分类 fallback 口径同步
- P2-02：Admin bundle 拆包优化

## 四、优先级调整说明

### 4.1 提升为高优先级

#### P2-01 推荐问题策略升级

原因：

- H5 首页入口用户可见。
- FAQ 数量增长后，问题会越来越明显。
- 与后续“场景化入口”“运营推荐”直接相关。

#### P2-03 检索低置信度兜底

原因：

- 误答比不答更影响信任。
- 当前问题不是“匹配不到”，而是“局部匹配过度自信”。
- 适合在 P2 做最小可信度收敛。

### 4.2 保持中优先级

#### P2-04 Admin 真实认证

原因：

- 临时登录不适合长期保留。
- 但 V1.1 不做多账号、多角色、多租户。
- P2 只应做单账号真实化、环境变量化或最小鉴权闭环。

#### P2-08 AI 配置后台化

原因：

- 这是后续 AI 问答质量迭代的基础。
- 但不建议在 P2 一口气做成完整 Prompt 管理平台。
- P2 更适合先完成配置模型、字段边界、管理入口和执行拆分判断。

### 4.3 保持低优先级或暂缓

#### P2-02 Admin bundle 拆包

原因：

- 当前是构建 warning，不是功能阻塞。
- 适合 P2 后段做，不应抢检索和推荐主线。

#### P2-06 Query Rewrite 验证

原因：

- 当前缺少自然样例，不宜为了证明而造假闭环。
- 等 KB 扩展或语义检索落地后再验证更客观。

## 五、P2 第一轮建议执行范围

如果 P2 要控制风险，建议第一轮只做以下 4 个方向：

```text
第一轮 P2 推荐范围：

1. P2-05 工程清理
2. P2-01 推荐策略升级
3. P2-03 检索低置信度兜底
4. P2-08 AI 配置后台化方案设计
```

第一轮不建议直接做：

```text
1. Admin/H5 视觉重构
2. 完整 Prompt 平台
3. 多账号、多租户、角色权限
4. RAG / 向量库 / embedding
5. 大规模目录重构
```

原因：

P2 的核心价值不是“做更多功能”，而是让 V1.1 的 FAQ 闭环更像一个可以演示、可以运营、可以继续迭代的产品骨架。

## 七、暂不纳入 P2 的事项

以下内容不进入 P2，避免范围漂移：

1. **完整 RAG / 向量库 / embedding 实装**
   - P2-06 只保留验证位，不实现向量库。

2. **多轮对话上下文记忆**
   - V1.1 继续保持单轮 FAQ 问答。

3. **用户注册 / 多账号 / 多角色 / 多租户**
   - P2-04 只做最小认证，不做完整权限系统。

4. **Admin + H5 视觉体系重构**
   - 这部分更适合进入 P3。
   - P2 可以保留必要交互微调，但不做整体 UI 风格重建。

5. **完整 Prompt 平台化**
   - P2 可以做 AI 配置后台化方案。
   - 但不做 Prompt 版本管理、灰度、多模型策略、效果评估平台。

6. **营养分析 / 菜单联动 / 个性化推荐**
   - 属于产品扩展方向，不属于 V1.1 FAQ 骨架深化。

7. **主后端语言迁移**
   - V1.1 继续保持当前 Node.js + Hono + TypeScript 路线。

## 九、结论

优化后的 P2 可以收敛为一句话：

> P2 重点不是扩功能，而是把 P1 跑通的 FAQ 闭环变得更可信、更可运营、更干净。

最终推荐主线：

```text
工程先清理
→ 推荐和检索先变可信
→ Admin 承接认证与 AI 配置管理
→ 性能和一致性后补
→ 向量检索相关验证延后
```

如果按这个顺序推进，P2 的范围会比原文更聚焦，也更符合当前 SmartDine “最小闭环、可验证、可复盘”的阶段目标。
