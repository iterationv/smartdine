# SmartDine 指令包复盘与阻塞处理规范

## 1. 背景

### 1.1 本轮 P0 执行范围

本轮复盘覆盖最近几轮 SmartDine V1.1 P0 执行包，重点包括：

- 后端 TASK-04 / TASK-05：问题日志、未命中记录、`/chat` 接入 retrieve 链路
- H5 执行包：TASK-08 / TASK-10
- Admin 执行包：TASK-06 / TASK-07 / TASK-09
- 跨端遗留问题收口包：H5 请求层收敛、Admin PATCH CORS 修复、三端回归
- TASK-12：初始知识条目数据录入与命中率验收
- TASK-11：热门问题统计模块
- P0 总体验收与后续指令设计过程中暴露的流程问题

### 1.2 为什么需要做这次复盘

这轮 P0 的主要问题并不集中在“代码不会写”，而集中在“执行方式是否稳定”。同样的技术能力，如果任务粒度、验收口径、联调判断、遗留收口和 git 清理规则不稳定，就会出现以下后果：

- 同一功能反复验收、反复解释
- 已完成功能仍然留下临时兼容逻辑
- 浏览器和命令行的验证结果相互打架
- 运行期日志、`dist`、临时样本容易污染工作区
- git 历史虽然可用，但不够贴合“执行包级”目标

这次复盘的目标不是再总结“做了什么”，而是把反复出现的流程问题提炼成后续默认规则。

---

## 2. 本轮执行中暴露出的核心问题

## 2.1 问题一：执行粒度和提交粒度被混用

### 问题名称

执行包级任务被默认按细粒度实现节奏推进，导致开发步骤和提交边界容易混在一起。

### 具体表现

- 后端 TASK-04 / TASK-05 阶段，route、service、save log、retrieve 接入被拆成多次细步提交。
- 到 H5 包、Admin 包、遗留问题收口包、TASK-11、TASK-12 时，用户开始反复强调“这是执行包级任务，不要拆成多个超细提交”。
- 中间允许 build、联调、临时修正，但用户希望这些都收敛在一个功能包里，而不是再形成多个依赖型 commit。

### 在哪些执行包中出现

- 后端 TASK-04 / TASK-05
- H5 TASK-08 / TASK-10
- Admin TASK-06 / TASK-07 / TASK-09
- 遗留问题收口包
- TASK-11 / TASK-12

### 为什么会出现

- P0 Spec 模板天然偏向“每条指令只做一件事”，更适合单步实现，不天然适合“执行包级闭环”。
- `GIT_RULES.md` 强调“一个 commit 只做一类事”，但没有进一步明确“用户明确要求执行包时，默认一个功能包完成后再提交”。
- AI 容易沿用前面细步任务的节奏，直到用户显式改口为“不要拆”。

### 造成的影响

- 一个完整链路容易被拆成多个相互依赖的 commit。
- 用户需要多次重复“不要中途 commit”。
- 功能开发、联调验证、最终提交之间的边界变模糊。
- git 历史可读，但不完全贴合“功能包级”验收节奏。

### 是否已解决

部分解决。后续几个执行包已经按“整包完成后再提交”的方式执行，但这条规则还没有沉淀成仓库长期默认规则。

### 后续建议

- 当用户明确说“执行包级”“功能包级”时，默认一次做完整闭环：实现、build、联调、清理、总结、最后提交。
- 执行粒度可以细，但提交粒度必须收敛到功能包。
- “中间联调”和“最终 commit”必须明确分离。

---

## 2.2 问题二：验收指令经常只写结果，不写前置条件和构造方式

### 问题名称

验收要求写了“应该看到什么”，但没写“如何构造这个场景”，导致执行时需要临时补方法。

### 具体表现

- H5 包要求验证 loading、error、未命中、空状态，但 error 场景需要临时决定如何 mock 或制造失败。
- TASK-11 统计页要验证 Top 10 和未命中数量，但如果当前日志不足，就必须临时补真实 `/chat` 样本数据。
- 总体验收类指令容易写“验证某任务仍完成”，但不说明是直接验证还是通过链路侧面确认。
- 如果不先确认真实路由和前缀，就容易把路径写错误判成接口失败。

### 在哪些执行包中出现

- H5 TASK-08 / TASK-10
- Admin TASK-06 / TASK-07 / TASK-09
- TASK-11
- 遗留问题收口包
- P0 总体验收指令设计

### 为什么会出现

- 验收指令通常偏结果导向，没有明确“前置数据、鉴权方式、错误态构造、空态构造、真实路由前缀”。
- 不同端的验证方式差异大，尤其浏览器态和命令行态并不等价。

### 造成的影响

- 验收动作重复设计，浪费时间。
- 容易出现“功能没问题，但验收方式不完整”的假阻塞。
- 一些任务虽然已经链路侧面验证通过，但仍需要额外解释是否真正完成。

### 是否已解决

未系统解决，主要靠每个执行包临时补说明。

### 后续建议

- 执行包模板中应默认写清：
  - 路由和接口前缀
  - 鉴权前置条件
  - 空状态、错误态、未命中态的构造方式
  - 若需要真实样本，如何通过接口生成而不是手改文件
  - 哪些任务是直接验证，哪些任务是链路侧面确认

---

## 2.3 问题三：联调阻塞很容易被误判成业务失败

### 问题名称

真实联调中的环境类问题、CORS 问题、服务存活问题，容易在第一时间被误判成“代码逻辑错了”。

### 具体表现

- Admin 知识状态切换中，PowerShell 直接调 `PATCH` 成功，但真实浏览器被 PATCH 预检拦截，最终定位为 CORS 而不是 knowledge 业务逻辑错误。
- 统计页、H5、Admin 的联调，都依赖真实端口和存活进程；如果服务中途退出或端口口径混用，很容易先怀疑代码。
- 浏览器与命令行的失败面不同：浏览器会先暴露 Origin、预检、页面鉴权，命令行更接近纯接口调用。

### 在哪些执行包中出现

- Admin TASK-06 遗留问题
- 遗留问题收口包
- TASK-11

### 为什么会出现

- 当前 AGENTS 虽然定义了阻塞分类模板，但没有把“浏览器失败 + 命令行成功”这种高频模式固化为默认诊断顺序。
- 一些前端验收天然依赖真实浏览器，不是 `curl` 或 PowerShell 结果可以完全替代的。

### 造成的影响

- 容易修改错误层级。
- 会把联调问题当业务 bug 修，增加返工。
- 如果只做命令行验证，可能错误地宣布“闭环完成”。

### 是否已解决

本轮已通过遗留问题收口包解决具体案例，但诊断顺序还没有写进长期规则。

### 后续建议

- 浏览器失败但命令行成功时，优先排查 CORS、Origin、预检、前端鉴权、路由前缀。
- 命令行失败但服务曾启动成功时，优先排查端口、进程、环境切换、Header。
- 前端联调类任务默认要求浏览器和命令行都验证，而不是二选一。

---

## 2.4 问题四：请求层和契约收敛不彻底，临时兼容容易变长期债务

### 问题名称

为了先把页面跑通而加入的临时兼容逻辑，如果不马上收口，就会形成架构层面的隐性返工。

### 具体表现

- H5 在 TASK-08 / TASK-10 完成后，`chatStore.js` 仍保留了直连 `/chat` 的 fallback、直接读 `VITE_API_SECRET`、在 store 里自行推断 `source`。
- 页面表现看起来已经达标，但请求层并没有真正统一到 `api/chat.js`。
- 这类问题最终只能在“跨端遗留问题收口包”里专门修一次。

### 在哪些执行包中出现

- H5 TASK-08 / TASK-10
- 遗留问题收口包

### 为什么会出现

- 当时优先目标是把产品页 UI 和主要交互做出来，临时兼容逻辑帮助快速联调。
- 原始任务对 `postChat()` 契约收敛要求不够彻底，给了 store 继续兜底的空间。

### 造成的影响

- API 层和 store 层职责边界变模糊。
- 契约规范分散到多个文件里。
- 以后如果后端契约再变，维护成本增加。

### 是否已解决

已解决。遗留问题收口包中已经统一回到单一 `postChat()` 入口。

### 后续建议

- 允许临时兼容，但必须明确标记为“短期桥接逻辑”。
- 这类桥接逻辑必须在后续最近一个收口包中移除，不能长期保留。
- API 契约归一化应在 API 层完成，不能散落到 store / page。

---

## 2.5 问题五：运行期日志、dist、临时样本会持续污染工作区

### 问题名称

真实联调天然会写日志、产出构建目录、制造测试样本，如果不主动清理，就非常容易混入 commit。

### 具体表现

- `questionLogs.json`、`missedQuestions.json` 在 `/chat` 联调和统计页样本生成时会被真实写入。
- `dist` 会在 build 或启动后反复出现。
- TASK-11 为了验证热门问题聚合，必须通过真实 `/chat` 生成样本日志；这些日志本身不应该进入源码提交。

### 在哪些执行包中出现

- 后端 TASK-04 / TASK-05
- 遗留问题收口包
- TASK-11
- TASK-12

### 为什么会出现

- P0 验收强调真实链路，而真实链路就会产生运行期副产物。
- 仓库当前不是“构建结果与运行日志完全隔离”的结构。

### 造成的影响

- `git status` 容易出现大量噪音。
- 提交边界容易被污染。
- 一旦误提交运行期文件，git 历史的可信度下降。

### 是否已解决

每个执行包最后都做了清理，但仍是“靠人记住”，不是规则驱动。

### 后续建议

- 所有联调和 build 完成后，默认执行一次 `git status --short`。
- 运行期 JSON、`dist`、临时样本必须在 commit 前恢复到 `HEAD`。
- 如果需要制造样本，只能通过真实接口写入，不能手改日志文件。

## 2.6 问题六：Windows / PowerShell 中文请求体验证存在噪音

### 问题名称

命令行验证在中文请求体、中文响应展示场景下存在编码噪音，容易把显示问题误判成接口问题。

### 具体表现

- PowerShell 直接发中文 JSON 时，容易出现乱码或服务端接收到的文本不稳定。
- `Invoke-RestMethod` 展示中文响应时，终端可能出现乱码，但实际 JSON 文件和浏览器 UI 正常。
- 统计页联调中，需要通过 UTF-8 文件或 `curl.exe --data-binary` 才能稳定生成样本。

### 在哪些执行包中出现

- `/chat` 联调阶段
- 遗留问题收口包
- TASK-11

### 为什么会出现

- Windows PowerShell 默认编码和部分命令的入参/出参展示不稳定。
- 命令行显示层和实际文件内容不完全一致。

### 造成的影响

- 容易误判为接口返回乱码或日志文件损坏。
- 验收步骤会被中文编码噪音干扰。

### 是否已解决

已通过临时技巧绕开，但不需要把具体命令都写入长期规则。

### 后续建议

- 长期规则只保留原则：PowerShell 中文场景优先使用 UTF-8 文件或 Unicode 转义 JSON。
- 具体命令写法保留在复盘文档或执行包中，不必塞进 AGENTS。

---

## 2.7 问题七：遗留问题如果不先收口，会拖慢下一包

### 问题名称

一个执行包已经“基本可用”，不代表它已经“完全闭环”；遗留问题如果不先收口，会在下一包继续拖后腿。

### 具体表现

- H5 产品化包做完后，页面看起来可用，但请求层仍有临时兼容。
- Admin 管理包做完后，知识状态切换在真实浏览器里仍被 PATCH CORS 拦截。
- 这两个问题如果不先单独收口，后面的 TASK-11、TASK-12、总体验收都会继续带着债务往下走。

### 在哪些执行包中出现

- H5 TASK-08 / TASK-10 之后
- Admin TASK-06 / TASK-07 / TASK-09 之后
- 遗留问题收口包

### 为什么会出现

- 某些执行包先把“主要可见功能”达成，就容易把结构问题和联调问题延后。
- 仓库规则里还没有明确“跨端遗留优先收口，再进入下一大包”。

### 造成的影响

- 任务状态出现“已完成但还有明显遗留”的模糊描述。
- 后续包需要先解释旧问题，再做新功能。
- 总体验收时容易再次打开已关闭任务。

### 是否已解决

本轮已通过专门的“遗留问题收口执行包”解决。

### 后续建议

- 一旦发现跨端遗留，优先做收口包。
- 收口包只做去兼容、补联调、补清理，不叠加新业务功能。
- 执行包完成后若还存在结构性遗留，应明确标注“基本完成”而不是“完全闭环”。

---

## 2.8 问题八：总体验收容易重复确认，缺少“直接验证 / 间接验证”映射

### 问题名称

后续大包、收口包和总体验收中，部分前置任务其实已被链路验证覆盖，但如果不显式说明，就会反复确认。

### 具体表现

- TASK-11、遗留收口、P0 总体验收设计中，用户多次要求明确说明：
  - 哪些任务仍保持完成态
  - 哪些功能是本轮直接验证
  - 哪些功能是通过完整链路侧面确认
- 例如：
  - `/chat` 契约是否未受影响
  - `/knowledge`、`/missed` 是否仍可访问
  - H5 之前的状态展示是否没有退化

### 在哪些执行包中出现

- 遗留问题收口包
- TASK-11
- P0 总体验收指令设计

### 为什么会出现

- 当前执行包强调“整包输出”，但没有规定输出里必须标明“直接验证 / 间接验证 / 未验证”。
- 用户更关心“这轮改动是否影响以前完成的任务”，而不是只看本次新增功能。

### 造成的影响

- 同一事实需要重复解释。
- 验收结论不够结构化。
- 影响总体验收效率。

### 是否已解决

本轮主要靠最终输出时人工补充，没有形成模板规则。

### 后续建议

- 跨任务回归和总体验收时，输出中默认增加三类标记：
  - 直接验证
  - 链路侧面确认
  - 本轮未覆盖

## 3. 问题分级

## 3.1 A 类：必须写进 AGENTS 的长期规则

这些问题反复出现，且不依赖当前 P0 阶段，应该固化为默认执行规则：

1. 执行粒度和提交粒度分离，执行包优先整包闭环后再提交
2. 验收前先确认真实路由、端口、前缀、鉴权和构造方式，不默认假设
3. 浏览器验收和命令行验收要同时考虑，联调失败先判环境/CORS/预检
4. 临时兼容逻辑必须在后续收口包中移除，不得长期保留
5. 运行期日志、`dist`、临时样本不进 commit，提交前必须清理
6. 遗留问题优先收口，再进入下一大包
7. 总体验收或跨包回归时，要区分直接验证和间接验证

## 3.2 B 类：更适合写进执行包模板补充，不一定进 AGENTS

这些问题很重要，但更适合放在任务说明模板里，而不是变成所有任务都强制执行的长期规则：

1. 明确列出每个错误态的构造方式
2. 明确是否需要通过真实 `/chat` 生成样本数据
3. 明确 Admin 验证时的登录前置条件
4. 明确统计页、未命中页、命中率验收的示例问题集合
5. 明确本轮联调要恢复哪些具体运行期文件

原因：

- 它们强依赖具体任务目标。
- 放进 AGENTS 会让长期规则变得过细、过长。

## 3.3 C 类：只作为阶段性复盘保留，不进入长期规则

这些内容适合保留在复盘文档中，但不适合进入 AGENTS：

1. “PATCH CORS 缺少 `PATCH`”这一条具体 bug 本身
2. “H5 store 曾经保留 fallback fetch”这一条具体实现细节本身
3. “统计页样本使用哪些具体问题”这一组具体题目
4. “PowerShell 里哪条命令乱码”这种命令级案例
5. 当前 P0 每个任务包的具体 commit 哈希和文件清单

原因：

- 它们是本轮案例，不是长期规则。
- AGENTS 应该管“以后默认怎么做”，不应该记录一次性的历史细节。

---

## 4. 推荐解决方案

## 4.1 以后如何避免执行粒度失控

- 先判断这次任务是“单步实现”还是“执行包闭环”。
- 用户明确要求“执行包”时，默认工作顺序为：
  1. 读文档和环境检查
  2. 连续修改相关文件
  3. build / 联调 / 回归
  4. 清理运行期副产物
  5. 先输出变更总结
  6. 最后一次性 commit
- 除非用户明确要求拆分，否则不把同一功能包拆成多个依赖型 commit。

## 4.2 以后如何避免验收反复解释

- 验收前先确认：
  - 真实接口前缀
  - 真实页面路由
  - 当前环境端口
  - 鉴权方式
  - 样本数据是否足够
- 对空态、错误态、未命中态、无数据态这类“非自然发生”的验收项，必须先写清构造方式再执行。
- 总体验收输出中应固定增加：
  - 本轮直接验证项
  - 通过链路侧面确认项
  - 本轮未覆盖项

## 4.3 以后如何减少联调误判

- 真实浏览器失败但命令行成功，优先查 CORS、Origin、预检、登录态。
- 命令行失败时先确认服务是否监听、端口是否正确、当前是否仍在 AI 测试环境。
- Windows 中文场景优先用 UTF-8 文件或 Unicode 转义 JSON，不直接把终端乱码当成接口 bug。

## 4.4 以后如何控制架构层临时债务

- 临时兼容逻辑允许存在，但要满足两个条件：
  - 只服务当前执行包联调
  - 后续最近一个收口包必须移除
- API 契约归一化必须收敛在 API 层或服务层，不向 store / page 扩散。

## 4.5 以后如何保证 git 历史质量

- 所有 build、联调、样本生成完成后，默认执行一次 `git status --short`。
- 运行期文件、`dist`、临时样本默认不提交。
- 真实接口写出来的日志文件可以用于验证，但提交前必须恢复。

## 4.6 以后如何安排遗留问题收口

- 功能包完成后如果发现：
  - 临时兼容未去掉
  - 真实浏览器链路未闭环
  - 某端验证口径和另一端冲突
  则下一步优先做遗留问题收口包，而不是继续叠加新功能。

---

## 5. 适合沉淀到 AGENTS 的规则草案

以下内容适合转成 AGENTS 的精炼规则：

1. 任务粒度规则：
   - 用户明确要求“执行包/功能包”时，默认整包闭环后再提交
   - 执行粒度和提交粒度必须分离
2. 提交规则：
   - 中间 build / 联调 / 临时兼容步骤不单独提交
   - 运行期日志、`dist`、临时样本不进 commit
3. 验收规则：
   - 验收前先确认真实路由、接口前缀、端口、鉴权和样本前置条件
   - 错误态、空态、未命中态必须先明确构造方式
   - 总体验收要说明直接验证和间接验证
4. 联调规则：
   - 浏览器验收和命令行验收都要考虑
   - 浏览器失败但命令行成功时，优先判定为联调问题而非业务失败
5. 收口规则：
   - 已知遗留优先收口，再进入下一大包
   - 临时兼容逻辑必须在收口包中移除

---

## 6. 一句话结论

这轮最关键的教训不是“代码难写”，而是“执行包必须先把粒度、验收口径、联调判定和清理规则讲清楚”，否则同样的功能会因为流程问题反复返工。
