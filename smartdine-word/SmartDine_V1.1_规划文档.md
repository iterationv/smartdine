- 版本：V1.1
- 阶段：P1 产品骨架完善
- 状态：P0 已完成，当前进入 P1
- 上游：V1 已完成最小流程验证（H5问答 + 后端处理 + FAQ/AI回答 + Admin基础配置）

---

> 阶段说明：P0 已完成并通过总体验收，含命中率验收通过；当前执行主线已切换至 P1。

## 一、阶段定位

V1 解决了"能不能跑"的问题，V1.1 解决"能不能用"的问题。

**一句话目标：** 将 SmartDine 从最小可运行原型，升级为具备稳定问答能力、基础知识运营能力和前台服务承接能力的产品雏形。

三条主线：
1. **问答更稳** —— 从能回答，到稳定命中并合理输出
2. **后台能管** —— 从可配置，到可运营
3. **前台像产品** —— 从测试页，到用户服务入口

---

## 二、当前问题判断

| 层面 | 问题 |
|------|------|
| 问答 | 自然语言适应性弱，不同问法命中不稳，过度依赖 FAQ 硬匹配 |
| 后台 | 无日志沉淀，无未命中回收，无热门统计，无知识优化入口 |
| 前台 | 无引导型入口，回答展示弱，缺加载/空/错误状态 |
| 工程 | 后端无分层，前端无状态管理，数据模型不规范 |

---

## 三、核心能力链路

```
知识沉淀 → 知识命中 → 受控回答 → 后台运营 → 前台承接
```

知识条目是 V1.1 的核心数据资产，所有能力围绕它构建。

---

## 四、版本范围

### P0：产品骨架（已完成）

**目标：** 问答链路做稳 + 最小知识运营闭环

当前状态：P0 已完成并通过总体验收，本文保留 P0 目标与验收口径作为阶段归档参考。

#### 知识条目结构升级
- FAQ 升级为知识条目，支持：title / question / answer / aliases / tags / status / updatedAt
- 完成旧 FAQ 兼容或迁移
- 录入 ≥20 条覆盖高频场景的初始知识条目（P0 验收测试数据基础）

#### 基础命中能力
- 关键词命中
- 同义问法 / 别名命中
- 问题基础清洗（去语气词、统一表达、别名映射）
- 简单排序逻辑

#### 问题日志 + 未命中记录
- 记录：提问内容 / 命中结果 / 最终回答 / 命中方式 / 时间
- 未命中问题独立入库，支持查询

#### 未命中回退策略（默认行为）
- 命中失败 → AI 兜底回答 + 未命中入库
- 统一回答前缀："这个问题暂时没有找到准确答案，以下仅供参考："

#### Admin 最小运营能力
- 知识条目管理页（增删改查 + 状态切换）
- 未命中问题列表（可查看 + 基础筛选）
- 热门问题基础统计（按频次）

#### H5 基础产品化
- 推荐问题 / 常见问题入口
- 回答卡片化展示
- 加载态 / 空状态 / 未命中提示 / 基础错误提示

#### 工程基础
- 后端基础分层：路由层 / 服务层 / AI 编排层 / 数据访问层
- AI 模块拆分：retrieve / matchKnowledge / buildContext / generateAnswer / saveQuestionLog
- 前端引入 Pinia，统一状态规范（见工程约束）
- 核心数据模型规范化

**P0 验收标准：**
- 预设 20 个高频问题，命中率 ≥ 80%
- 未命中问题可在 Admin 查看，延迟 ≤ 5 分钟
- 知识条目可完成新增 / 编辑 / 状态切换，无报错
- H5 具备推荐入口和基础状态展示

## 五、工程约束

### 不改变主技术架构
保留现有 Node 主后端 + Vue3 前端，不迁移，不重构，只做必要增强。

### 允许的工程增强
1. **前端引入 Pinia** —— H5 和 Admin 统一状态规范：全局状态（问答态、用户态）走 store，页面局部状态用 `ref/reactive`，禁止过度全局化
2. **后端基础分层** —— 补充路由 / 服务 / AI 编排 / 数据访问四层，目标是减少未来迁移成本，不是重写
3. **数据模型规范化** —— 知识条目 / 问题日志 / 未命中问题的结构必须在 P0 收敛清楚

### 技术演进节奏
- V1.1：产品骨架，必要工程增强
- V1.2 / V1.3：骨架完善，能力闭环
- V1.4：评估正式技术升级（Java 迁移 / Python AI 服务 / 基础设施重构）

**技术升级必须服务于产品节奏，不先于产品建设发生。**
